## 16. JavaScript Engines & Internals

### ✅ What is a JavaScript Engine

A **JavaScript engine** is a program that reads, compiles, and executes JavaScript code. Each browser ships its own engine:

| Engine | Used by |
|--------|---------|
| **V8** | Chrome, Node.js, Edge |
| **SpiderMonkey** | Firefox |
| **JavaScriptCore (Nitro)** | Safari |

All engines follow the same general pipeline: **Parse → Compile → Execute**.


### ✅ Parsing

Before any code runs, the engine reads the source text and turns it into a structured format it can work with.

#### Step 1 — Tokenization (Lexing)

The source string is broken into **tokens** — the smallest meaningful units.

```js
let x = 5 + 3;

// Tokens:
// [let] [x] [=] [5] [+] [3] [;]
```

#### Step 2 — AST (Abstract Syntax Tree)

Tokens are assembled into a **tree structure** that represents the code's meaning.

```js
// let x = 5 + 3;
//
// VariableDeclaration
//   └── VariableDeclarator (x)
//         └── BinaryExpression (+)
//               ├── Literal (5)
//               └── Literal (3)
```

The AST discards whitespace and syntax characters — only semantics remain. Tools like Babel, ESLint, and Prettier all work by transforming ASTs.

#### Eager vs Lazy Parsing

- **Eager parsing** — fully parses and builds AST immediately.
- **Lazy parsing** — does a quick scan of functions not yet called (just checks syntax). Full parse happens when the function is actually called.

This speeds up startup time — most functions aren't called on page load.


### ✅ Compilation Pipeline

Modern JS engines don't just interpret — they **compile to machine code** for performance.

#### Interpreter vs Compiler

| | Interpreter | Compiler |
|---|---|---|
| How | Executes line by line | Translates all code first, then runs |
| Startup | Fast | Slow (compile time) |
| Execution speed | Slower | Faster |

#### JIT Compilation (Just-In-Time)

V8 uses a **JIT compiler** — a hybrid approach:

1. Start with the **interpreter** for fast startup.
2. Profile code as it runs — identify **hot code** (code that runs repeatedly).
3. Progressively move hot code through faster compiler tiers.
4. If assumptions break — **deoptimise** back to a lower tier.

#### V8 Compilation Tiers (current — 4 tiers)

```
Source Code
    ↓
  Parser  →  AST
    ↓
Ignition (interpreter)      →  Bytecode         — fast startup, collects type feedback
    ↓  (getting warm)
Sparkplug (baseline JIT)    →  Machine Code      — fast compile, no speculative opts
    ↓  (getting hot)
Maglev (mid-tier optimizer) →  Machine Code      — 10× faster than TurboFan to compile
    ↓  (very hot)
TurboFan (peak optimizer)   →  Optimised Machine Code — highest performance
    ↓  (if assumptions break)
Deoptimise  →  back to lower tier
```

| Tier | Role | Trade-off |
|------|------|-----------|
| Ignition | Interpreter | Fastest startup, slowest execution |
| Sparkplug | Baseline JIT | Fast compile, no type-based opts |
| Maglev | Mid-tier JIT | Good perf without TurboFan's compile cost |
| TurboFan | Peak JIT | Slowest to compile, fastest to run |


### ✅ V8 Internals — Ignition & TurboFan

#### Ignition (Interpreter)

- Converts AST to **bytecode** — a compact, platform-independent instruction set.
- Executes bytecode immediately — fast startup, lower memory than machine code.
- Collects **type feedback** as code runs — which types does each variable hold?

```js
function add(a, b) {
  return a + b;
}

add(1, 2);   // Ignition notes: a=number, b=number
add(3, 4);   // confirms: always numbers
```

#### TurboFan (Optimising Compiler)

- Takes hot bytecode + type feedback from Ignition.
- **Assumes** types stay the same (e.g. `a` is always a number).
- Generates highly optimised machine code based on those assumptions.

```js
// TurboFan can generate a fast integer-add instruction
// because it knows both args are always numbers
add(5, 6);   // runs as optimised machine code — very fast
```

#### The pipeline together

```
AST → Ignition → Sparkplug → Maglev → TurboFan
                     ↑ deopt back to lower tier if assumptions break
```


### ✅ Hidden Classes

V8 uses **hidden classes** (also called **shapes** or **maps**) to optimise property access on objects.

#### The problem

JavaScript objects are dynamic — you can add/remove properties at runtime. Naively, every property lookup would be a hash table search. V8 avoids this with hidden classes.

#### How hidden classes work

When you create an object and add properties, V8 assigns a **hidden class** and creates transitions between classes as properties are added.

```js
function Point(x, y) {
  this.x = x;  // hidden class C0 → C1 (added x)
  this.y = y;  // hidden class C1 → C2 (added y)
}

const p1 = new Point(1, 2); // hidden class C2
const p2 = new Point(3, 4); // reuses hidden class C2 — same shape!
```

Because `p1` and `p2` share a hidden class, V8 knows `x` is always at the same memory offset — property access is as fast as a C struct.

#### What breaks hidden classes

Adding properties in a **different order** creates different hidden class chains — V8 can't share them.

```js
const a = {};
a.x = 1;
a.y = 2;
// hidden class: {} → {x} → {x,y}

const b = {};
b.y = 2;
b.x = 1;
// hidden class: {} → {y} → {y,x}  — different chain!
```

**Best practice:** Always initialise all properties in the same order — preferably in the constructor.

#### Deleting properties

`delete obj.prop` also breaks the hidden class chain — the object transitions to a new class and can lose optimisation.

```js
const obj = { x: 1, y: 2 };
delete obj.x; // breaks hidden class — sets obj to "dictionary mode" (slow)

// Prefer setting to null or undefined instead
obj.x = null;
```


### ✅ Inline Caching

**Inline caching (IC)** speeds up repeated property lookups by remembering the result of a previous lookup.

#### How it works

The first time V8 accesses `obj.x`, it does a full lookup and records the hidden class + property offset. The next time the same code runs on an object with the **same hidden class**, V8 skips the lookup entirely.

```js
function getX(obj) {
  return obj.x; // IC records: "for hidden class C2, x is at offset 8"
}

getX(p1); // full lookup — IC is empty (uninitialized)
getX(p2); // IC hit — skips lookup, reads directly from offset 8
getX(p3); // IC hit — same hidden class → instant access
```

#### IC states

| State | Meaning |
|-------|---------|
| **Uninitialized** | First call — no info yet |
| **Monomorphic** | Always same hidden class — fastest |
| **Polymorphic** | 2–4 different hidden classes — still fast |
| **Megamorphic** | Many different hidden classes — falls back to slow path |

```js
function readName(obj) {
  return obj.name;
}

readName({ name: "Atik" });          // monomorphic
readName({ name: "Bob", age: 30 }); // polymorphic — different shape
readName(new Person("Atik"));        // megamorphic if too many shapes
```

**Best practice:** Call functions with objects of the same shape to stay monomorphic.


### ✅ Deoptimisations

When an optimising tier's assumptions are violated, V8 **deoptimises** — discards the optimised machine code and falls back to a lower tier.

#### Lazy Unlinking (current behaviour)

V8 no longer immediately removes deoptimised functions during GC. Instead, it marks the function and defers cleanup to the **next time it is called** — this avoids GC pauses and improves performance significantly.

```
Function deoptimised → marked_for_deoptimization flag set
    ↓  (next invocation)
V8 checks flag → jumps to CompileLazyDeoptimizedCode → re-compiles from lower tier
```

#### What triggers deoptimisation

```js
function add(a, b) {
  return a + b;
}

add(1, 2);      // TurboFan optimises: assumes a, b are always numbers
add(3, 4);      // fast — machine code
add("x", "y"); // type changed! — deoptimise, back to bytecode
```

#### Common deoptimisation triggers

| Trigger | Why |
|---------|-----|
| Changing variable types | Breaks type assumptions |
| Adding properties after init | Breaks hidden class |
| `delete` on object properties | Forces dictionary mode |
| `arguments` object usage | Prevents some optimisations |
| `try/catch` around hot code | Limits optimisation scope |
| Very large functions | TurboFan may skip them |

#### Writing optimisation-friendly code

```js
// Bad — type changes
function process(val) {
  if (typeof val === "string") return val.length;
  return val * 2; // same function, different types
}

// Better — separate functions per type
function processString(val) { return val.length; }
function processNumber(val) { return val * 2; }
```

```js
// Bad — properties added after construction
const obj = {};
obj.x = 1;
obj.y = 2;

// Good — all properties in constructor
function Point(x, y) {
  this.x = x;
  this.y = y;
}
```
