## 11. Event Loop & Concurrency

### ✅ Stack, Heap, Queue

#### Call Stack

The **call stack** is where JavaScript tracks function execution. It follows **LIFO** (Last In, First Out) — the last function pushed is the first to pop off.

```js
function greet(name) {
  return `Hello, ${name}`;
}

function sayHello() {
  const message = greet("Atik");
  console.log(message);
}

sayHello();
```

**Stack frames (in order):**

```
1. sayHello()     → pushed
2. greet("Atik")  → pushed on top
3. greet()        → returns, popped
4. console.log()  → pushed, then popped
5. sayHello()     → popped (stack is now empty)
```

#### Stack Overflow

Happens when the call stack exceeds its limit — always caused by unbounded recursion.

```js
function infinite() {
  return infinite(); // RangeError: Maximum call stack size exceeded
}
```

#### Heap

The **heap** is an unstructured memory region where **objects, arrays, and functions** are stored. Variables hold references (pointers) to values in the heap — not the values themselves.

```js
const user = { name: "Atik" }; // object lives in heap, 'user' is a reference
const arr  = [1, 2, 3];        // array lives in heap
```

| Data | Where stored |
|------|-------------|
| Numbers, strings, booleans, `null`, `undefined` | Stack (by value) |
| Objects, arrays, functions | Heap (by reference) |

#### Queue (Callback Queue / Task Queue)

The **queue** holds callbacks that are ready to run. They only move to the call stack when it is **completely empty**.

```js
console.log("A");
setTimeout(() => console.log("B"), 0); // queued, not run immediately
console.log("C");

// Output: A  C  B
```

Even with `0ms` delay, `"B"` runs last — it must wait for the stack to clear.


### ✅ Web APIs Layer

JavaScript itself has no timers, no DOM, no network. These are provided by the **browser (or Node.js runtime)** as Web APIs. When you call `setTimeout`, `fetch`, or add a DOM event listener, the browser takes over — JavaScript just registers the callback and moves on.

#### How it works

```
JS Code → calls setTimeout(fn, 1000)
             ↓
         Browser Web API tracks the timer
             ↓  (1000ms passes)
         Callback fn pushed to Macrotask Queue
             ↓  (stack is empty)
         Event Loop moves fn to Call Stack
             ↓
         fn executes
```

#### Common Web APIs

| API | Provided by |
|-----|------------|
| `setTimeout` / `setInterval` | Browser timer API |
| `fetch` | Browser network API |
| DOM events (`click`, `scroll`) | Browser event system |
| `localStorage` | Browser storage API |
| Geolocation, Clipboard | Browser device APIs |

This is why JavaScript can be **non-blocking** despite being single-threaded — heavy work is offloaded to the browser while JS moves on to the next line.


### ✅ How the Event Loop Works

The **event loop** is a continuous process that connects the call stack, web APIs, and queues.

#### The Full Picture

```
   ┌──────────────────────────────────────────┐
   │           JavaScript Engine              │
   │   ┌─────────────┐    ┌───────────────┐   │
   │   │ Call Stack  │    │     Heap      │   │
   │   └──────┬──────┘    └───────────────┘   │
   └──────────┼───────────────────────────────┘
              │ empty?
              ▼
   ┌──────────────────────┐
   │   Microtask Queue    │  ← Promise .then, queueMicrotask, MutationObserver
   └───────────┬──────────┘
               │ all drained?
               ▼
   ┌──────────────────────┐
   │  Macrotask Queue     │  ← setTimeout, setInterval, DOM events, I/O
   └───────────┬──────────┘
               │ pick one task
               ▼
          Call Stack
```

#### Event Loop Cycle (step by step)

1. Run all **synchronous code** on the call stack.
2. Stack is empty → drain the **entire microtask queue** (all of them).
3. Pick **one macrotask**, push it to the stack, run it.
4. Stack is empty again → drain **microtasks** again.
5. Repeat forever.

```js
console.log("1 - sync");

setTimeout(() => console.log("2 - macrotask"), 0);

Promise.resolve().then(() => console.log("3 - microtask"));

console.log("4 - sync");

// Output:
// 1 - sync
// 4 - sync
// 3 - microtask
// 2 - macrotask
```


### ✅ Job Queue vs Callback Queue

#### Callback Queue (Macrotask Queue)

- Holds callbacks from older or heavier async APIs.
- The event loop picks **one** macrotask per cycle.

**Sources:**

| Source | Example |
|--------|---------|
| `setTimeout` | `setTimeout(fn, 0)` |
| `setInterval` | `setInterval(fn, 1000)` |
| DOM events | click, scroll, keydown |
| `MessageChannel` | `postMessage` |
| `setImmediate` | Node.js only |
| I/O callbacks | File/network (Node.js) |

#### Job Queue (Microtask Queue)

- Has **higher priority** than the macrotask queue.
- The event loop drains the **entire queue** before touching any macrotask.

**Sources:**

| Source | Example |
|--------|---------|
| Promise `.then` / `.catch` / `.finally` | `Promise.resolve().then(fn)` |
| `async/await` continuations | code after `await` |
| `queueMicrotask()` | `queueMicrotask(fn)` |
| `MutationObserver` | DOM change callbacks |

#### Priority in action

```js
setTimeout(() => console.log("macrotask"), 0);

Promise.resolve()
  .then(() => console.log("microtask 1"))
  .then(() => console.log("microtask 2"));

// Output:
// microtask 1
// microtask 2
// macrotask
```

`setTimeout` was registered first — but both microtasks run before it.

#### Microtask starvation

If microtasks keep adding more microtasks, macrotasks can be **starved** (never run).

```js
function flood() {
  Promise.resolve().then(flood); // infinite microtask loop
}
flood(); // macrotasks will never execute
```


### ✅ Macro vs Micro Tasks

#### Summary Table

| Feature | Microtask | Macrotask |
|---------|-----------|-----------|
| Queue | Microtask Queue / Job Queue | Task Queue / Callback Queue |
| Priority | Higher | Lower |
| When it runs | After current task, before next macrotask | One per event loop cycle |
| Draining | Entire queue at once | One at a time |
| Examples | Promise `.then`, `queueMicrotask` | `setTimeout`, DOM events |

#### Complex output prediction example

```js
console.log("start");

setTimeout(() => {
  console.log("timeout 1");
  Promise.resolve().then(() => console.log("micro inside timeout"));
}, 0);

Promise.resolve()
  .then(() => {
    console.log("micro 1");
    setTimeout(() => console.log("timeout 2"), 0);
  })
  .then(() => console.log("micro 2"));

setTimeout(() => console.log("timeout 3"), 0);

console.log("end");
```

**Output:**

```
start
end
micro 1
micro 2
timeout 1
micro inside timeout
timeout 2
timeout 3
```

**Why:**

1. Sync: `start`, `end`
2. Microtasks drain: `micro 1` (registers `timeout 2`) → `micro 2`
3. Macrotask: `timeout 1` (registers `micro inside timeout`)
4. Microtasks drain: `micro inside timeout`
5. Macrotask: `timeout 2`
6. Macrotask: `timeout 3`

#### `async/await` under the hood

`await` is syntactic sugar over Promises — every line **after** `await` becomes a microtask continuation.

```js
async function run() {
  console.log("A");
  await Promise.resolve();
  console.log("B"); // resumes as a microtask
}

console.log("1");
run();
console.log("2");

// Output:
// 1
// A
// 2
// B
```

`"B"` runs after `"2"` because `await` yields and the current sync code (`"2"`) finishes first.


### ✅ `queueMicrotask()`

A direct way to schedule a microtask without creating a Promise.

```js
console.log("start");

queueMicrotask(() => console.log("microtask"));

console.log("end");

// Output:
// start
// end
// microtask
```

#### When to use it over `Promise.resolve().then()`

- Slightly more explicit and performant (no Promise object created).
- Useful when you need microtask timing but don't have a Promise-based API.

```js
function notify(callback) {
  queueMicrotask(callback); // runs after current task, before any macrotask
}
```


### ✅ `requestAnimationFrame`

`requestAnimationFrame` (rAF) schedules a callback to run **before the next browser repaint** — roughly every 16ms (60fps).

#### Where it sits in the loop

```
Macrotask → Microtasks → requestAnimationFrame → Browser Paint → next Macrotask
```

It is neither a microtask nor a standard macrotask — it runs in its own **rendering step** before paint.

```js
function animate() {
  // update something on screen
  element.style.left = `${position++}px`;
  requestAnimationFrame(animate); // schedule next frame
}

requestAnimationFrame(animate);
```

#### Why not `setTimeout` for animations?

| | `setTimeout(fn, 16)` | `requestAnimationFrame` |
|---|---|---|
| Timing | Approximate, can drift | Synced to display refresh rate |
| Paused when tab hidden | No | Yes (saves battery/CPU) |
| Smooth animation | Less reliable | Optimal |


### ✅ Concurrency vs Parallelism

#### Concurrency

Multiple tasks **make progress** over time, but not necessarily at the same instant. JavaScript achieves this via the event loop — switching between tasks by offloading async work.

```
Time →
Task A: ██░░░░██░░░░██
Task B: ░░██░░░░██░░░░
(alternating — concurrent, not parallel)
```

#### Parallelism

Multiple tasks run **at the exact same time** on separate threads/cores.

JavaScript is **single-threaded** — it cannot do true parallelism in the main thread. However it can achieve parallelism via **Web Workers** (browser) or **Worker Threads** (Node.js).

```js
// Web Worker — runs in a separate thread
const worker = new Worker("worker.js");
worker.postMessage({ data: [1, 2, 3] });
worker.onmessage = (e) => console.log("Result:", e.data);
```

#### Summary

| | Concurrency | Parallelism |
|---|---|---|
| Definition | Tasks interleave over time | Tasks run simultaneously |
| JavaScript main thread | Yes (via event loop) | No |
| JavaScript with Workers | — | Yes |


### ✅ Blocking the Main Thread

JavaScript is single-threaded — if the call stack is busy with a long-running task, **nothing else can run**: no UI updates, no event handlers, no timers.

#### Example of blocking

```js
// This freezes the browser for ~3 seconds
const start = Date.now();
while (Date.now() - start < 3000) {
  // busy waiting — blocks everything
}
console.log("Done");
```

During those 3 seconds: clicks are ignored, the page can't scroll, animations freeze.

#### How to avoid blocking

| Technique | How |
|-----------|-----|
| Break work into chunks | Use `setTimeout(fn, 0)` to yield between chunks |
| Use `requestAnimationFrame` | Schedule work in render frames |
| Web Workers | Offload CPU-heavy work to a separate thread |
| `scheduler.postTask()` | Modern API for prioritized task scheduling |

#### Chunking example

```js
const items = new Array(1_000_000).fill(0);
let index = 0;

function processChunk() {
  const end = Math.min(index + 10000, items.length);
  for (; index < end; index++) {
    // process items[index]
  }
  if (index < items.length) {
    setTimeout(processChunk, 0); // yield to event loop between chunks
  }
}

processChunk();
```


### ✅ Node.js Event Loop Phases

Node.js has its own event loop with distinct **phases** — each phase has its own queue.

```
   ┌─────────────────────────────────┐
   │           timers                │  ← setTimeout, setInterval callbacks
   └──────────────┬──────────────────┘
                  ↓
   ┌─────────────────────────────────┐
   │        pending callbacks        │  ← I/O errors deferred from last cycle
   └──────────────┬──────────────────┘
                  ↓
   ┌─────────────────────────────────┐
   │        idle, prepare            │  ← internal use only
   └──────────────┬──────────────────┘
                  ↓
   ┌─────────────────────────────────┐
   │            poll                 │  ← fetch new I/O events, execute I/O callbacks
   └──────────────┬──────────────────┘
                  ↓
   ┌─────────────────────────────────┐
   │            check                │  ← setImmediate callbacks
   └──────────────┬──────────────────┘
                  ↓
   ┌─────────────────────────────────┐
   │        close callbacks          │  ← socket.on('close', ...)
   └─────────────────────────────────┘
```

#### `process.nextTick` vs `setImmediate`

- `process.nextTick` — runs **before** the next event loop phase (even before microtasks in older Node versions, now same priority as microtasks).
- `setImmediate` — runs in the **check phase**, after I/O callbacks.

```js
setImmediate(() => console.log("setImmediate"));
process.nextTick(() => console.log("nextTick"));
Promise.resolve().then(() => console.log("promise"));

// Output (Node.js):
// nextTick
// promise
// setImmediate
```

#### `setTimeout` vs `setImmediate` (in I/O context)

```js
const fs = require("fs");

fs.readFile("file.txt", () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));
});

// Inside an I/O callback, setImmediate always runs before setTimeout
// Output:
// immediate
// timeout
```


### ✅ Common Interview Gotchas

#### Gotcha 1 — `setTimeout` inside a loop with `var`

**With `let` — works as expected:**

```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 1);
}
// Output: 0  1  2
```

`let` is block-scoped — each iteration gets its **own separate `i`**:

```js
// Iteration 1
{ let i = 0; setTimeout(() => console.log(i), 1); } // i = 0

// Iteration 2
{ let i = 1; setTimeout(() => console.log(i), 1); } // i = 1

// Iteration 3
{ let i = 2; setTimeout(() => console.log(i), 1); } // i = 2
```

**With `var` — broken:**

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 1);
}
// Output: 3  3  3
```

`var` is function-scoped — there is only **one shared `i`**. By the time the callbacks run, the loop has already finished and `i` is `3`:

```js
var i; // declared once, shared across all iterations

i = 0; setTimeout(() => console.log(i), 1); // will log 3
i = 1; setTimeout(() => console.log(i), 1); // will log 3
i = 2; setTimeout(() => console.log(i), 1); // will log 3

// loop ends: i becomes 3
```

#### Gotcha 2 — Promise inside `setTimeout`

```js
setTimeout(() => {
  Promise.resolve().then(() => console.log("micro"));
  console.log("macro");
}, 0);

// Output:
// macro
// micro
```

Inside a macrotask, microtasks still run before the next macrotask.

#### Gotcha 3 — Multiple `await` calls

```js
async function main() {
  console.log("A");
  await null;
  console.log("B");
  await null;
  console.log("C");
}

console.log("1");
main();
console.log("2");

// Output:
// 1
// A
// 2
// B
// C
```

Each `await` yields — sync code (`"2"`) runs, then `"B"`, then `"C"` in subsequent microtask turns.

#### Gotcha 4 — `Promise.resolve` vs `new Promise`

```js
console.log("start");

new Promise((resolve) => {
  console.log("executor"); // runs synchronously
  resolve();
}).then(() => console.log("then"));

console.log("end");

// Output:
// start
// executor
// end
// then
```

The Promise **executor** runs synchronously. Only `.then()` is a microtask.


### ✅ Quick Reference

```
Sync code → Microtasks (all) → rAF → Paint → One Macrotask → Microtasks (all) → ...
```

| What you write | Queue |
|---|---|
| Normal code | Call Stack (sync) |
| `Promise.then()` | Microtask |
| `async/await` resume | Microtask |
| `queueMicrotask()` | Microtask |
| `MutationObserver` | Microtask |
| `process.nextTick` | Microtask (Node.js) |
| `requestAnimationFrame` | Render step (before paint) |
| `setTimeout` / `setInterval` | Macrotask |
| DOM event listeners | Macrotask |
| `setImmediate` | Macrotask / Check phase (Node.js) |
